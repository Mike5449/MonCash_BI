5. STACK TECHNOLOGIQUE  (2 min)
--------------------------------------------------------------------------------

FRONTEND
  - React 19 + TypeScript + Vite (bundler moderne, ultra-rapide en dev)
  - React Query v5 (@tanstack/react-query) pour la gestion du cache client
  - React Router 7 pour la navigation SPA
  - Recharts pour les graphiques, lucide-react pour les icones
  - Design system maison base sur des design tokens CSS

BACKEND
  - Python 3.11 + FastAPI (framework async performant, OpenAPI natif)
  - SQLAlchemy + Databricks SQL Connector pour l'acces aux donnees
  - Pydantic Settings pour la configuration
  - slowapi pour le rate limiting
  - JWT (python-jose) + bcrypt / passlib pour l'auth
  - pytesseract + mrz + Pillow pour l'OCR des cartes d'identite

CACHE & PERFORMANCE
  - Redis (cache partage multi-workers, TTL par fraicheur)
  - Cache en memoire en fallback si Redis indisponible

DONNEES
  - Databricks SQL warehouse (Azure)
  - Tables gr_dgc_dwh_prd.ods_dl.mfs_mobile_banking_audit,
    ht_location, etc.

DEPLOIEMENT
  - Docker + Docker Compose (3 conteneurs : backend, frontend / nginx, redis)
  - Nginx en reverse proxy + serveur de statique
  - Caddy en frontal pour TLS / HTTPS automatique (Let's Encrypt)


--------------------------------------------------------------------------------
6. ARCHITECTURE — POINTS-CLES  (2 min)
--------------------------------------------------------------------------------

  - Une seule entree reseau (port 443) — backend et Redis internes au
    reseau Docker

  - Same-origin — nginx proxie /api/* -> backend, pas de CORS a gerer

  - Read-only cote Databricks — aucune ecriture, uniquement des SELECT

  - Trois conteneurs Docker isoles :
      * frontend  : nginx + SPA React (static)
      * backend   : FastAPI + Python
      * redis     : cache partage

  - Les containers communiquent via un reseau Docker prive interne

7. RECOMMANDATIONS POUR LA MISE EN PRODUCTION  (5 min)
--------------------------------------------------------------------------------

============================
RECOMMANDATION 1 · AZURE SERVICE PRINCIPAL (auth Python)
============================

"Actuellement, le backend utilise DefaultAzureCredential d'Azure Identity, qui
essaie plusieurs methodes en cascade : token en cache, Azure CLI local, browser
interactif. AUCUNE de ces methodes ne marche dans un container Linux en
production — pas d'interface graphique, pas d'Azure CLI installe, pas
d'utilisateur logue."

La solution : creer un Service Principal Azure AD dedie a l'application,
avec juste les permissions necessaires pour acceder au Databricks SQL
warehouse.

Ce qu'il faut demander a l'equipe Azure / IT :
  1. Creation d'une App Registration dans Azure AD
  2. Generation d'un client secret (valable 24 mois, a faire tourner)
  3. Attribution des permissions Databricks (user_impersonation)
  4. Ajout du SP dans les ACL du SQL warehouse

3 variables a mettre dans .env :
  AZURE_TENANT_ID
  AZURE_CLIENT_ID
  AZURE_CLIENT_SECRET

DefaultAzureCredential les detecte automatiquement — aucun changement de
code necessaire.

Alternative simple : Databricks Personal Access Token (PAT) — 1 seule
variable, mais liee a un utilisateur.


============================
RECOMMANDATION 2 · SERVEUR LINUX DEDIE
============================

Specifications minimales requises :

  Ressource     Spec
  ---------     ----
  OS            Ubuntu Server 22.04 LTS ou 24.04 LTS
  CPU           4 vCPU (x86_64)
  RAM           8 GB
  Disque        100 GB SSD
  Reseau        100 Mbps symetrique
  GPU           Aucun (sauf si on veut du LLM local)

Ports a ouvrir (entrant) :
  22   - SSH (cle uniquement, pas de mot de passe)
  80   - Redirect HTTPS
  443  - HTTPS

Sortie autorisee :
  *.azuredatabricks.net:443         (requetes SQL)
  login.microsoftonline.com:443     (Azure AD tokens)
  api.anthropic.com:443             (futur chatbot)
  registry-1.docker.io:443          (pull images Docker)

Logiciels sur l'hote :
  - Docker Engine 24+
  - Docker Compose plugin 2.20+
  - Caddy (TLS automatique via Let's Encrypt)
  - UFW (firewall)
  - fail2ban (protection brute-force SSH)
  - unattended-upgrades (patches securite auto)


============================
CE QU'IL FAUT FOURNIR A L'EQUIPE DEVOPS POUR LANCER
============================

  [ ] Un VM Linux avec les specs ci-dessus
  [ ] Un nom de domaine (ex. bi.moncash.digicel.ht) + acces DNS
  [ ] Le Service Principal Azure (3 credentials)
  [ ] Acces Git (SSH deploy key ou personal token)
  [ ] Autorisation firewall pour les domaines sortants

Une fois ces prerequis en place, le deploiement se resume a 3 commandes :

  git clone <repo> /opt/moncash-bi
  cp .env.example .env && vim .env
  docker compose up -d --build
